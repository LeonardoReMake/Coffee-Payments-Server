Необходимо реализовать страницу отображения заказа и его текущего статуса.
Пользователь будет попадать на эту страницу автоматически после завершения оплаты в платёжной системе.

# Получение заказа
 - Страница получает идентификатор заказа (order_id) из query-параметров URL.
 - На основе order_id необходимо загрузить всю информацию о заказе.

# Информация, отображаемая на странице
Страница должна содержать следующую информацию:
 - Логотип клиента — Device.logo_url.
 - Адрес кофемашины (Device.location).
 - Название напитка.
 - Размер напитка — маленький / средний / большой.
 - Цена напитка.
 - Описание текущего статуса заказа (см. раздел 3).
 - Информация для клиента, задаваемая в Device.* полях для соответствующего статуса (может содержать HTML).

# Отображение статусов заказа
В зависимости от значения Order.status страница должна отображать соответствующее состояние и набор UI-элементов.

## Статус: pending
 - Состояние: загрузка
 - Описание: "Проверяем оплату заказа..."
 - Информация для клиента: Device.client_info_pending (HTML поддерживается)

## Статус: paid
 - Состояние: успешное
 - Описание: "Заказ успешно оплачен, начинаем готовить"
 - Информация для клиента: Device.client_info_paid (HTML поддерживается)

## Статус: not_paid
 - Состояние: ошибка
 - Описание: "Оплата не прошла"
 - Кнопка "Повторить оплату"
    - По нажатию должен создаваться новый запрос на создание платежа (аналогично логике на экране order_info_screen).
 - Информация для клиента: Device.client_info_not_paid (HTML поддерживается)

## Статус: make_pending
 - Состояние: загрузка
 - Описание: "Готовим напиток..."
 - Информация для клиента: Device.client_info_make_pending (HTML поддерживается)

## Статус: successful
 - Состояние: успешное
 - Описание: "Напиток готов"
 - Информация для клиента: Device.client_info_successful (HTML поддерживается)

# Обновление статуса заказа
 - Страница должна проверять статус каждую секунду отправляя REST запрос каждую секунду.
 - При получении нового статуса интерфейс должен обновляться без перезагрузки страницы.
 - UI переходы должны быть плавными и соответствовать UX-стилю проекта.

# Требования к UI/UX
 - Страница создаётся по принципу mobile first и должна быть полностью адаптивной.
 - Все визуальные элементы (состояние загрузки, успешное состояние, ошибочное состояние, кнопки, описания) должны соответствовать правилам из CONSTITUTION.md, включая:
    - использование централизованного файла пользовательских текстов ошибок;
    - отсутствие технических ошибок в UI;
    - корректное отображение HTML-тегов в тексте для клиента.
